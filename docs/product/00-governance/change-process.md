# 需求变更流程

## 0. 不可变约束
- 所有变更必须满足 P0 原则：业务请求仅可访问去中心化 RPC。
- 任何试图引入中心化后端业务请求的变更必须直接驳回。
- 任何试图引入遥测、追踪脚本或用户数据库依赖的变更必须直接驳回。

## 1. 变更触发
满足任一条件即触发变更：
- 新增功能。
- 修改现有流程或业务规则。
- 调整性能、安全、稳定性目标。

## 2. 变更步骤
1. 先完成 P0 原则符合性检查（通过/不通过）。
2. 在 `02-requirements/` 更新需求条目。
3. 在 `03-user-stories/` 增补或修订故事与验收条件。
4. 若涉及术语/状态变化，更新 `05-domain/`。
5. 在 `04-roadmap/` 调整版本计划。

## 3. 变更准入检查
- 是否说明业务价值。
- 是否定义边界条件。
- 是否具备可执行的验收标准。
- 是否识别潜在风险（安全/性能/体验）。
- 是否满足 P0 原则（仅 RPC 业务依赖）。

## 4. 完成定义（Definition of Done）
- 文档已更新且可追踪到需求 ID。
- 代码实现与测试通过。
- 回归测试覆盖关键链路。
- 网络请求审查结果显示无非 RPC 业务请求。
